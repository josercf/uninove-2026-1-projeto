SEMANA 15 – Padrões de Projeto (GoF)

O que são Design Patterns?

  • Definição: Soluções reutilizáveis para problemas comuns em design de software
  • Origem: Gang of Four (GoF) - Erich Gamma, Richard Helm, Ralph Johnson, John Vlissides (1994)
  • 23 Padrões clássicos: Divididos em 3 categorias
  • Vantagem: Código mais limpo, testável, manutenível e escalável
  • Comunicação: Linguagem comum entre desenvolvedores
Objetivo: Entender QUANDO usar cada padrão, não apenas decorar nomes

15 – Semana 15

Criacionais: Criação de Objetos

Singleton
Um único objeto global. Ex: Logger, Database connection
Factory Method
Interface para criar objetos. Delega a criação.
Abstract Factory
Criar famílias de objetos relacionados
Builder
Construção passo a passo. Objetos complexos.

Prototype: Clonar objetos existentes ao invés de criar novos

15 – Semana 15

Exemplo: Singleton

// TypeScript class Logger { private static instance: Logger; private constructor() {} static getInstance(): Logger { if (!Logger.instance) { Logger.instance = new Logger(); } return Logger.instance; } log(message: string): void { console.log(`[LOG] ${message}`); } } // Uso const logger = Logger.getInstance(); logger.log("Aplicação iniciada");

Benefício: Garante uma única instância. Útil para recursos compartilhados (database, config).

15 – Semana 15

Estruturais: Composição de Classes

Adapter
Transformar interface incompatível
Facade
Interface simples para subsistema complexo
Decorator
Adicionar comportamento dinamicamente
Proxy
Representante/substituto de outro objeto

Outros: Bridge, Composite, Flyweight

15 – Semana 15

Exemplo: Facade

// Subsistema complexo com múltiplas classes class Database { connect() {} } class Logger { log() {} } class Cache { set() {} } // Facade - interface simples class ApplicationFacade { private db = new Database(); private logger = new Logger(); private cache = new Cache(); startApplication() { this.db.connect(); this.logger.log("DB connected"); this.cache.set("app_started", true); } } // Cliente só precisa conhecer a Facade const app = new ApplicationFacade(); app.startApplication();

Benefício: Encapsula complexidade, facilita manutenção.

15 – Semana 15

Comportamentais: Comunicação entre Objetos

  • Strategy: Encapsular algoritmos intercambiáveis
  • Observer: Notificar múltiplos objetos sobre mudanças
  • Command: Encapsular solicitação como objeto
  • Template Method: Definir esqueleto de algoritmo em classe base
  • State: Mudar comportamento quando estado muda
  • Iterator: Acessar elementos de coleção sequencialmente

Outros (5): Chain of Responsibility, Interpreter, Mediator, Memento, Visitor

15 – Semana 15

Exemplo: Strategy Pattern

// Interface para estratégias interface PaymentStrategy { pay(amount: number): void; } // Implementações concretas class CreditCardPayment implements PaymentStrategy { pay(amount: number) { console.log(`Pago ${amount} com Cartão Crédito`); } } class PixPayment implements PaymentStrategy { pay(amount: number) { console.log(`Pago ${amount} via Pix`); } } // Contexto usa a estratégia class ShoppingCart { constructor(private strategy: PaymentStrategy) {} checkout(amount: number) { this.strategy.pay(amount); } } // Uso const cart = new ShoppingCart(new PixPayment()); cart.checkout(100);

Benefício: Algoritmo pode ser mudado em runtime sem alterar cliente.

15 – Semana 15

Quando NÃO Usar Padrões

  • Over-Engineering: Usar padrão complexo para problema simples
  • Anti-Pattern: God Object (classe gigante), Spaghetti Code
  • Premature Optimization: Implementar padrão antes de ter requisito
  • Falta de Contexto: Aplicar padrão porque está "na moda"
Regra de Ouro: Use padrões quando eles resolvem um problema REAL, não quando "parece legal"
