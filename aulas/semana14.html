SEMANA 14 – Modelagem de Dados: NoSQL (MongoDB & Redis)

SQL vs NoSQL

Característica SQL (Relacional) NoSQL
Schema Estruturado e rígido Flexível e dinâmico
Escalabilidade Vertical (mais CPU/RAM) Horizontal (mais servidores)
Transações ACID (seguro) Eventual consistency
Quando usar Dados estruturados, relacionados Dados variáveis, grande volume, alta velocidade
Melhor abordagem: Combinar ambas! Poliglota = SQL + NoSQL + Cache

MongoDB – Conceitos Básicos

  • Database: Banco (análogo ao banco SQL)
  • Collection: Tabela (sem schema obrigatório)
  • Document: Registro em formato JSON/BSON (não relacional)
  • Field: Campo dentro do documento (propriedade JSON)
  • Índice: Para otimizar buscas e consultas
  • Aggregation Pipeline: Processamento de dados complexo
Vantagem: Flexibilidade
Vantagem: Performance leitura
14 – Semana 14

MongoDB: Embedded vs Referenced

Embedded (Desnormalizado)

{ _id: 1, nome: "João", endereco: { rua: "Rua A", cidade: "São Paulo" } }

Quando: Dados relacionados, poucos acessos separados

Referenced (Normalizado)

// usuarios {_id: 1, nome: "João"} // enderecos {usuario_id: 1, rua: "Rua A"}

Quando: Dados reutilizáveis, muitos acessos independentes

Dica: MongoDB favorece desnormalização. Use embedded quando possível para melhor performance.

Schema Design Patterns

  • Subset Pattern: Armazenar apenas dados frequentemente acessados no documento principal
  • Extended Reference: Copiar campos frequentes para evitar JOINs
  • Bucket Pattern: Agrupar dados por ranges (timestamps, locações geográficas)
  • Attribute Pattern: Usar arrays de key-value para atributos dinâmicos
Exemplo Subset: Armazenar últimas 5 compras com o usuário
14 – Semana 14

Redis: Estruturas de Dados

  • String: Valor simples (texto, números) - cache básico
  • Hash: Objeto com múltiplos campos - usuário, sessão
  • List: Coleção ordenada - fila, timeline, logs recentes
  • Set: Coleção não ordenada, valores únicos - tags, followers
  • Sorted Set: Set ordenado por score - ranking, leaderboard
  • Stream: Log de eventos com consumidores - pub/sub persistente
Vantagem Redis: Operações em memória, velocidade extremamente alta (microsegundos)
14 – Semana 14

Redis: Casos de Uso

Cache
Armazenar resultados de queries frequentes com TTL
Session Storage
Armazenar dados de sessão do usuário
Rate Limiting
Contar requisições por IP/usuário em janelas de tempo
Pub/Sub
Mensagens em tempo real entre aplicações

Outros: Queues (RabbitMQ alternativa), Real-time analytics, Leaderboards

MySQL
