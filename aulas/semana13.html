SEMANA 13 – Modelagem de Dados: SQL (MySQL)

Banco de Dados Relacional

  • Conceito: Dados organizados em tabelas (relações) com linhas e colunas
  • Tabelas: Cada tabela representa uma entidade (usuários, pedidos, produtos)
  • Colunas: Atributos/propriedades (id, nome, email, data_criacao)
  • Linhas: Registros individuais (um usuário por linha)
  • Chaves: Primária (identifica unicamente), Estrangeira (relaciona tabelas)
Vantagem: Integridade de dados
Vantagem: Consultas complexas
Vantagem: ACID transactions

Diagrama ER – Componentes

  • Entidades: Retângulos representando objetos do mundo real (Usuário, Pedido)
  • Atributos: Círculos/Elipses conectadas às entidades (nome, email, data)
  • Chave Primária: Atributo sublinhado que identifica unicamente
  • Relacionamentos: Losangos mostrando como as entidades se relacionam
  • Cardinalidade: Notação indicando a multiplicidade (1:1, 1:N, N:N)
Exemplo: Uma Pessoa pode ter vários Endereços (1:N), um Endereço pertence a uma Pessoa

Tipos de Relacionamento

Tipo Notação Exemplo
Um para Um (1:1) Pessoa — Documento Uma pessoa tem um CPF único
Um para Muitos (1:N) Pessoa —< Endereço Uma pessoa pode ter vários endereços
Muitos para Muitos (N:N) Aluno >—< Disciplina Um aluno cursa várias disciplinas, e uma disciplina tem vários alunos
Nota: Relacionamentos N:N precisam de uma tabela de junção (tabela intermediária)
13 – Semana 13

Normalização de Dados

  • 1ª Forma Normal (1FN): Dados atômicos (não repetir colunas), sem grupos repetitivos
  • 2ª Forma Normal (2FN): Atender 1FN + todos os atributos dependem da chave primária completa
  • 3ª Forma Normal (3FN): Atender 2FN + sem dependências transitivas entre atributos não-chave
  • Objetivo: Reduzir redundância, melhorar integridade, evitar anomalias
Problema: Redundância
Solução: Normalização
Resultado: Integridade
13 – Semana 13

Chaves e Índices

  • Chave Primária (PK): Identifica unicamente cada registro. Não pode ser NULL. Uma por tabela.
  • Chave Estrangeira (FK): Referencia a chave primária de outra tabela. Mantém integridade relacional.
  • Chave Única (UNIQUE): Garante unicidade, mas permite NULL. Múltiplas possíveis por tabela.
  • Índice: Estrutura de dados para acelerar buscas. Deve ser usado em colunas frequentemente consultadas.
Atenção: Cada índice usa espaço e lentifica inserts/updates. Use com moderação!

DDL: CREATE TABLE

CREATE TABLE usuarios ( id INT PRIMARY KEY AUTO_INCREMENT, email VARCHAR(255) NOT NULL UNIQUE, nome VARCHAR(100) NOT NULL, data_criacao TIMESTAMP DEFAULT CURRENT_TIMESTAMP, ativo BOOLEAN DEFAULT TRUE, INDEX idx_email (email) ); CREATE TABLE pedidos ( id INT PRIMARY KEY AUTO_INCREMENT, usuario_id INT NOT NULL, total DECIMAL(10,2) NOT NULL, status ENUM('pendente', 'confirmado', 'entregue'), FOREIGN KEY (usuario_id) REFERENCES usuarios(id) ON DELETE CASCADE );

Tipos: INT, VARCHAR, DECIMAL, TIMESTAMP, BOOLEAN, ENUM, TEXT, JSON

Exemplo: SaaS com Multi-Tenancy

CREATE TABLE tenants ( id INT PRIMARY KEY AUTO_INCREMENT, nome VARCHAR(100) NOT NULL, plano ENUM('free', 'pro', 'enterprise') ); CREATE TABLE usuarios ( id INT PRIMARY KEY AUTO_INCREMENT, tenant_id INT NOT NULL, email VARCHAR(255) NOT NULL, FOREIGN KEY (tenant_id) REFERENCES tenants(id), UNIQUE KEY uk_tenant_email (tenant_id, email) ); CREATE TABLE projetos ( id INT PRIMARY KEY AUTO_INCREMENT, tenant_id INT NOT NULL, nome VARCHAR(100), FOREIGN KEY (tenant_id) REFERENCES tenants(id) );
Multi-tenant: Cada tabela deve ter tenant_id para isolar dados de clientes diferentes
